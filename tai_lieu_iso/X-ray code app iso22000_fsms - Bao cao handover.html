X-ray mức code — app iso22000_fsms

Báo cáo handover · Repo github.com/mrhuychien/iso22000_fsms · nhánh main · commit 39597f8 · ngày 14/06/2026

Tóm tắt một đoạn. App là một hệ FSMS ISO 22000:2018 khá bài bản về khung quản lý: 18 module nghiệp vụ, wiring sự kiện (doc_events) và scheduler đầy đủ, chuỗi auto-create đúng tinh thần ISO (đánh giá→NCR, thu hồi→truy xuất, CCP vượt ngưỡng→leo thang), phân quyền dòng (row-level) cho NCR/Rủi ro được escape đúng cách, không phát hiện lỗ SQL injection hay endpoint cho khách (guest). Nhưng đúng như anh cảm nhận, phần đặc thù an toàn thực phẩm FMCG còn thiếu: allergen, EMP, phụ gia/nhãn, food defense đều vắng ở mức code, và PRP/SSOP mới làm được vệ sinh công nhân + vệ sinh khu vực. Ngoài ra có 1 nợ kỹ thuật đáng chú ý: bộ validator workflow đang rỗng (hook quảng cáo "OR-logic approval" thực chất chưa làm gì).

Chuỗi auto-create (đã xác minh trong source)

  • Audit Execution submit → tạo NCR từ findings.
  • Recall Event submit → tạo FSMS Traceability Trace.
  • NCR on_update_after_submit: khi workflow_state == "Chuyển tiếp" và chưa reissue → tự tạo NCR mới ("re-issued") qua new.insert(ignore_permissions=True) và back-link.
  • Purchase Receipt submit → tự tạo inspection log; CCP/Calibration/Material Inspection submit → handler leo thang.

Scheduler

daily: NCR quá hạn, thiết bị tới hạn hiệu chuẩn, mẫu tới hạn hủy, kiểm tra khẩn cấp tới hạn. weekly: tài liệu tới hạn review, nhắc kickoff kế hoạch đánh giá, digest tuân thủ đào tạo. monthly: nhắc review rủi ro, kiểm tra review năm của HACCP plan, nhắc diễn tập truy xuất. cron 0 8 1 12 *: nhắc đánh giá cuối năm (08:00 ngày 1/12).

Overrides & fixtures

  • override_whitelisted_methods: frappe.model.workflow.apply_workflow → workflow_validation.apply_workflow_with_validation. Không có override_doctype_class.
  • extend_bootinfo: boot.boot_fsms_session. jinja.methods: format_vnd, format_vn_date, signature_or_blank.
  • fixtures: Roles FSMS% + 4 role trưởng bộ phận; Custom Fields fsms_%; Property Setters (Core/Traceability/Supplier/Recall); toàn bộ Workflow State/Action Master; Workflows FSMS%; Notification/Print Format/Dashboard/Number Card/Workspace lọc theo module; Email Template FSMS%; và seed single-doctype (Document Category, NCR Source, Risk Score Reference, Supplier Category, Emergency Scenario).

3. PERMISSION MAP — phân quyền

  • permission_query_conditions: FSMS NCR → fsms_ncr.permissions.get_query_conditions; FSMS Risk Register → fsms_context_risk.permissions.get_query_conditions.
  • has_permission: FSMS NCR → fsms_ncr.permissions.has_permission (kiểm tra bằng ORM thuần).
  • Phạm vi dòng: user thường chỉ thấy NCR thuộc phòng ban (User Permission) của mình hoặc do mình tạo. PRIVILEGED_ROLES = {FSMS Manager, FSMS Director, FSMS Internal Auditor, System Manager} thấy toàn bộ.
Đánh giá an toàn: WHERE clause dựng bằng f-string nhưng mọi giá trị do người dùng kiểm soát đều qua frappe.db.escape() — đây là idiom đúng của Frappe cho permission_query_conditions (vốn phải trả về chuỗi, không thể tham số hóa). Không phải lỗ injection. 
return f"`tabFSMS NCR`.owner = {frappe.db.escape(user)}"
dept_list = ", ".join(frappe.db.escape(d) for d in departments)

Giới hạn của lần quét này: GitHub chặn rate-limit (HTTP 429) nên chưa quét hết api.py/permissions.py của cả 20 thư mục module. Kết luận bảo mật là đại diện và chắc chắn cho phần đã đọc (top-level api.py, fsms_ncr, fsms_traceability, fsms_recall, permissions.py), chưa phải quét vét cạn. Khi shell hồi phục, nên grep -rn "@frappe.whitelist\|allow_guest" apps/iso22000_fsms để chốt 100%.

5. Workflow — phát hiện quan trọng

Bộ validator workflow đang RỖNG. workflow_validation.py (24 dòng) chỉ là wrapper; registry FSMS_WORKFLOW_VALIDATORS rỗng (chỉ có 1 dòng comment mẫu). Nghĩa là logic "OR-logic approval" mà hook quảng cáo chưa được hiện thực — nó chỉ gọi thẳng core apply_workflow. Đây là hook "chết". Mọi ràng buộc duyệt hiện dựa hoàn toàn vào định nghĩa Workflow trong fixtures.
FSMS_WORKFLOW_VALIDATORS: dict[str, str] = {
    # "FSMS NCR": "iso22000_fsms.fsms_ncr.api.validate_workflow_transition",
}

State machine không nằm ở file này — nó được khai báo qua fixtures Workflow FSMS% (README nêu 28 workflow). Cần xác nhận từng workflow có đủ trạng thái/transition khi shell hồi phục.

9. RUNBOOK & bước tiếp theo

Khi shell (sandbox) hồi phục, để quét vét cạn phần còn lại:

git clone https://github.com/mrhuychien/iso22000_fsms
cd iso22000_fsms
grep -rn "@frappe.whitelist\|allow_guest" .          # quét hết API
find . -path "*/doctype/*/*.json" | wc -l            # đếm DocType thật
grep -rln "ignore_permissions" .                     # rà bypass quyền
ls iso22000_fsms/tests/                               # đối chiếu test

Backlog build đề xuất (thứ tự ưu tiên), mỗi mảng → 1-2 DocType mới + fixtures + doc_events:

  1. Allergen: ma trận allergen trên Item + DocType nhiễm chéo/đánh giá rủi ro allergen.
  2. PRP đầy đủ: tách thành DocType vận hành riêng cho Pest Control, Chemical/SDS, Glass & Brittle Plastic, Waste, Water/Utilities, Rework (thay vì enum trên PRP Item).
  3. EMP: kế hoạch + log swab bề mặt/không khí, gắn khu đóng gói.
  4. Phụ gia/nhãn: register phụ gia (TT 24/2019) + kiểm soát nhãn/quy cách (NĐ 111/2021, ghi nhãn allergen).
  5. Food defense: DocType TACCP/VACCP cho FSSC 22000.

Sau khi chốt backlog, chuyển sang quy trình build có kiểm soát (thiết kế DocType → fixtures → doc_events → test).

Phương pháp: do shell sandbox đang kẹt (không clone được) và raw.githubusercontent.com/api.github.com bị chặn allowlist, X-ray này đọc trực tiếp nội dung file nhúng trong trang github.com/blob (qua 2 subagent). Các kết luận "0 match" được kiểm trên toàn file blueprint. Phần chưa đọc được nêu rõ ở mục 4 & 8.
